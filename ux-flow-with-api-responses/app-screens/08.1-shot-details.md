## Screen Overview

- **Purpose**: Show full video shot details with metadata, engagement stats, and related content.
- **Primary user(s)**: `USER`
- **Entry points**: Tap a shot from Shots Feed or deep-link from comments/notifications
- **Exit / next screens**: Back to Shots Feed (`08-shots`) or navigate into another Shot Details screen via related item
- **Success criteria**: Shot video and metadata load successfully, with related shots displayed
- **Key constraints**: Must support fast media loading and avoid blocking UI on parallel API calls

## Common UI Rules

- **Submit protection**: Not applicable (no form submission on this screen)
- **Offline**: Show inline banner *"You're offline. Check your connection and try again."*
- **5xx**: Toast *"Something went wrong. Please try again."* and log error
- **Validation (`422`)**: Not applicable for this screen
- **Rate-limit (`429`)**: Read `Retry-After` and show inline countdown *"Try again in {N}s."*
- **Status mapping**:  
  `401` → force re-auth inline redirect  
  `403` → access denied toast  
  `404` → shot not found empty state  
  `5xx` → retry toast

## UX Flow

### Shot Details Initial Load

1. User taps a shot from the feed or comments section.
2. Screen opens with a loading skeleton.
3. In parallel, client triggers:
   - Shot details fetch → [`GET /shots/:shotId`](../modules/shots.md#85-get-shot-details)
   - Related shots fetch → [`GET /shots/:shotId/related`](../modules/shots.md#86-get-related-shots)

> **WHY parallel fetch?**  
> Parallel requests reduce perceived latency. User sees video + metadata + recommendations together instead of sequential loading delays.

4. UI renders:
   - Top: Video player (autoplay muted, paused until interaction if required by device policy)
   - Details: title, description, upload date, creator info
   - Stats: views, favorites, shares
   - Related section: horizontal scroll list of recommended shots

---

### Interaction Actions

1. **Favorite / Share**
   - Uses existing feed actions (shared endpoint from Shots module)
   - UI updates optimistically (immediate local update before server confirmation)

2. **Open Related Shot**
   - User taps related item
   - Screen reloads with new `shotId`
   - Triggers same parallel fetch flow

> **WHY soft navigation refresh?**  
> Keeps navigation stack clean and avoids deep nested routing loops inside media-heavy screens.

## Storage & Session

| Token | Storage | Lifetime | Cleared when |
| --- | --- | --- | --- |
| Access token | In-memory (or SecureStorage if persisted) | ~15 min (confirm from auth module) | Logout / refresh failure |
| Refresh token | SecureStorage — hardware-backed key vault (OS-level protected storage like Keychain or Android Keystore) | ~30 days (confirm) | Logout or token rotation failure |

> **WHY SecureStorage?**  
SecureStorage prevents token extraction even on rooted devices by isolating secrets in hardware-backed encryption.

## Edge Cases

### Shot Not Found
- **Trigger**: `404` from `GET /shots/:shotId`
- **UI response**: Empty state screen
- **Message**: *"This shot is no longer available."*
- **Action**: Back to Shots Feed
- **Note**: Happens when content is deleted or access revoked

### Related Shot Refresh Loop Risk
- **Trigger**: User repeatedly taps related shots rapidly
- **UI response**: Debounced navigation (block repeated requests for short window)
- **Message**: none
- **Action**: ignore duplicate taps within cooldown window
- **Note**: Prevents race condition (multiple overlapping requests returning out-of-order data)

## UX Audit

**Minor**
- Multiple parallel API calls may compete for network bandwidth on low-end devices  
  **Why**: concurrent video + JSON requests can delay first render  
  **Fix**: prioritize video metadata first, defer related shots by a short lazy-load window (e.g. 300–500ms)

- Related horizontal list may overflow visually on small screens  
  **Why**: long titles + thumbnails reduce readability  
  **Fix**: enforce fixed card width + truncate titles
